# 回声复赛备战计划（工程审查版）

更新时间：2026-06-28

目标：提交一个符合复赛要求、能运行、能解释大模型调用、材料齐全且策略上有竞争力的作品。

---

## 1. 赛事要求结论

复赛不是只交策划。赛事方明确要求：

1. 复赛作品需要初步开发且能运行
2. PPT 要在初赛基础上结合实际作品效果介绍
3. 演示视频建议 3 分钟左右，最长不超过 5 分钟
4. 海报必须竖版，70cm x 150cm，文件不低于 2MB
5. 可运行产品可以是 APK、快应用、链接、demo 等
6. 代码包必须包含核心功能代码，并重点标注大模型调用部分

评分权重：

- 创新性 40%：选题、功能、界面、交互
- 应用价值 30%：可行性、前景
- 完成度 20%：资料齐全、资料质量
- 大模型应用能力 10%：场景说明 + API 调用能力

因此当前策略必须同时满足：

- 能打开、能操作、能录制、能对话、能展示回忆
- 能证明代码里有真实大模型调用链路
- 能解释如果云端失败为什么产品仍然可用
- 能给评委一个稳定入口，而不是只依赖临时本地隧道

## 2. 当前方案判断

### 2.1 推荐最终方案

最终提交采用“三入口 + 一代码包”策略：

1. Sealos：可选云端验证入口，国内部署，配置 vivo AIGC 官网 AppKey（环境变量 `VIVO_APP_KEY`）后可额外验证真实云端大模型链路；不配置也不影响成品运行
2. Hugging Face：备用演示入口，只用于展示完整产品流程和本地兜底
3. 本地运行：评委下载代码包后双击 `install.bat`、`start.bat` 运行
4. 代码包：包含 Web、后端、Android APK 工程、端侧桥接参考、部署文档、自检脚本

### 2.2 方案优越性

这个方案比“只交 HF 链接”更强：

- HF 海外环境访问 `api-ai.vivo.com.cn` 容易超时，不能证明国内云端 API 可用
- Sealos 国内网络更适合打通 vivo API，若你后续愿意配置 AppKey，可通过 `/api/debug` 给评委直接验证
- 本地包保证即使线上部署异常，评委仍可运行产品

这个方案比“只交 APK”更稳：

- 复赛线上评审，评委未必愿意安装 APK
- Web 链接打开成本最低
- APK 工程和端侧桥接保留为 vivo 生态理解和决赛端侧化路线，而不是把复赛成败押在真机 SDK 联调上

这个方案比“只讲端侧规划”更符合评分：

- 大模型能力评分要求“可以通过 API 调用实现产品功能”
- 当前 Web + Flask 已经有 `call_bluelm_cloud()`、`api_debug()`、`api_chat()`
- 端侧化作为加分叙事，但不冒充已完成真机 NPU 实测

### 2.3 可行性判断

可行，但必须按优先级推进：

- P0：保证 Web 产品和代码包能运行
- P1：如果需要云端加分，再部署 Sealos 并通过 `/api/debug`
- P2：录制视频、PPT、海报
- P3：APK/端侧作为工程补充和决赛路线，不作为复赛主验收条件

不建议继续扩展新功能。复赛剩余时间应全部投入部署、材料质量、演示稳定性。

## 3. 本轮工程审查结果

### 3.1 已发现并修正的问题

1. `start.bat` 缺少 `setlocal EnableDelayedExpansion`
   - 后果：`!IP!` 可能无法正常展开
   - 已修正

2. `android-apk/app/src/main/java/com/echo/app/MainActivity.java` 传错构造参数
   - 原来传 `this`
   - `WebAppInterface` 构造函数需要 `WebView`
   - 已修正为 `new WebAppInterface(webView)`

3. Android JSBridge 前端调用方式错误
   - Java 端 `chat(String prompt, String callbackName)` 接收的是回调名字符串
   - 前端原来传 JS 函数对象，APK 环境下不可靠
   - 已修正为注册全局回调名，再传给 `window.BlueLM.chat(message, callbackName)`

4. `.gitignore` 未忽略临时生成物
   - 已补 `tmp/`、`output/`

5. `.dockerignore` 未忽略生成物
   - 已补 `output/`、`tmp/`

6. README、部署文档、自检文档存在口径风险
   - 已统一为：Sealos 主评审、HF 备用、本地包兜底
   - 端侧表述已改成“APK Demo 预留桥接，真机 NPU 需后续联调验收”

7. PPT prompt 过于保守，偏离计划设想
   - 已重写成“最终提交方案口径”
   - 同时保留条件：Sealos 未验收不得写已验证

### 3.2 本地接口自检结果

已使用 Flask test client 验证：

- `/test`：200
- `POST /api/voices`：200
- `POST /api/voices/<id>/upload`：200
- `GET /api/voices/<id>/model-status`：200
- `POST /api/chat`：200，未配置 Key 时走 `local-fallback`
- `/api/stats`：200
- `/api/debug`：200，未配置 Key 时返回 `not_configured`

说明：

- 本地核心产品链路可运行
- 当前还不能宣称“评委环境 API 一定有效”
- 只有 Sealos 配置真实 vivo AppKey 后，`/api/debug` 返回 `cloud_api.status = ok`，才算云端加分链路通过

## 4. 最终作品定位

### 4.1 对评委的核心表达

一句话：

> 回声是一款面向家庭陪伴场景的 AI 声音传承应用，用“声音档案 + 长辈人格对话 + 回忆沉淀”的方式，让亲人的声音不只被保存，还能继续回应。

技术表达：

> 复赛版本通过 Flask + SQLite + 原生 Web 完成可运行 Demo，本地语义引擎负责兜底，云端大模型调用由 `call_bluelm_cloud()` 实现并作为可选加分项，`api_debug()` 负责链路自检，APK 工程保留 BlueLM 3B 端侧化路径。

### 4.2 不要夸大的内容

不要写：

- 已完成真实语音克隆
- 已完成真机 NPU 推理实测
- HF 可以证明 vivo 国内 API 可用
- Sealos 未验收前就写“已验证云端大模型”

可以写：

- 已完成声音档案创建与录音上传
- 已完成 AI 对话链路
- 已完成云端 API 调用代码与自检接口
- 已完成本地兜底，确保演示不中断
- 已预留 APK 端侧桥接和决赛端侧化路线

## 5. 大模型代码定位

评委问“大模型在哪里”，直接指：

- `app.py`
  - `get_vivo_app_key()`：读取 vivo AIGC 官网 AppKey
  - `api_debug()`：验证云端 API 状态
  - `call_bluelm_cloud()`：调用 vivo 蓝心大模型平台 API
  - `generate_mock_reply()`：云端失败后的本地语义兜底
  - `generate_tts_audio()`：TTS 语音回复
  - `api_chat()`：对话主入口

- `static/js/chat.js`
  - `_callChatAPI()`：端云调度入口
  - `_callOnDevice()`：APK 端侧桥接入口
  - `_callCloudAPI()`：调用 Flask 云端路径

- `android-apk/app/src/main/java/com/echo/app/WebAppInterface.java`
  - 复赛 APK Demo 桥接，当前返回降级信号

- `android-bridge/WebAppInterface.java`
  - BlueLM 3B SDK 接入参考，决赛真机联调路线

补充：你提供的“端侧文本审核”属于 vivo 端侧 SDK 能力，适合放入决赛路线，用于长辈对话内容的本地安全审核。复赛当前不强行接入 AAR，避免新增 APK 构建和真机权限风险。
